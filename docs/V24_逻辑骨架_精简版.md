# V24 逻辑骨架精简版

更新时间: 2026-03-29
适用范围: 当前 `E:\APP\ELO` APP 实际代码

## 1. 目标

这份文档不描述全部功能细节，只定义系统最关键的逻辑骨架，用来回答 4 个问题:

1. 数据进入系统后，如何变成统一对象
2. 决策大脑到底输出什么
3. 推荐如何进入执行、结算和复盘
4. Gate 如何决定系统是否继续放行

核心原则:

- 决策大脑只负责判断，不负责 UI 和状态管理
- 执行层不重新推理，只消费标准预测结果
- 每个推荐都必须能被自动结算
- Gate 只基于结算结果工作，不直接依赖模型内部变量

## 2. 最小对象集

系统现在必须稳定维护 6 个核心对象。

### 2.1 `AppMatch`

作用: 比赛的标准化输入对象。

关键字段:

- `match_id`
- `source_id`
- `home_team`
- `away_team`
- `league`
- `match_date`
- `match_time`
- `odds_home`
- `odds_draw`
- `odds_away`
- `handicap_line`
- `source`

约束:

- 任何数据源进入模型前，必须先被转换为 `AppMatch`
- `AppMatch` 是模型层、执行层、结算层共享的唯一比赛载体

### 2.2 `PredictionBundle`

作用: 决策大脑的标准输出对象。当前代码中表现为 `predict_match()` 返回的字典。

至少包含:

## 3. 六层骨架

### 3.1 数据输入层

职责:

- 从 Titan 抓赛程、赔率、让球线、赛果
- 统一修复编码和时间字段
- 过滤异常比赛
- 输出 `AppMatch`

当前主源:

- `Titan007`

当前风险:

- 外部字段格式变化
- 反爬导致临时失效
- 历史赛事回查依赖 `source_id`

### 3.2 决策大脑层

职责:

- 基于 `Market + ELO + Poisson + XGBv0` 输出统一预测包
- 给出多玩法推荐和置信度
- 保证摘要结果一致

当前能力:

- 胜平负
- 让球胜平负
- 总进球
- 比分
- 半全场
- 多维指数

当前边界:

- 不负责展示
- 不负责快照管理
- 不负责结算

### 3.3 执行层

职责:

- 将 `PredictionBundle` 转成用户可用推荐
- 输出单场推荐
- 生成混合二串一推荐
- 落地快照
- 导出报告

当前关键规则:

- 二串一优先混合玩法
- 二串一腿来自高置信玩法候选
- 执行层不自己推导概率

### 3.4 结算层

职责:

- 自动回收完场比赛
- 生成单场结算
- 通过单场结算驱动二串一自动结算

当前结算对象:

- `SingleSettlement`
- `ParlayTicket(status=won/lost)`

当前边界:

- 结算只做“命中判定”
- 结算不重新生成推荐

### 3.5 Gate 风控层

职责:

- 统计命中率
- 统计 EV 偏差
- 跟踪连败
- 触发熔断

当前规则:

- `losing_streak >= 3` 触发 `breaker_on`
- 旧记录若缺少预期命中率，不纳入 EV 偏差计算

### 3.6 反馈闭环层

职责:

- 结算后更新 ELO
- 回流单场训练样本
- 刷新 Gate
- 更新二串一状态

闭环终点:

- 样本增长
- 风险状态更新
- 推荐是否继续放行

## 4. 主流程

### 4.1 赛前主流程

```text
Titan 抓比赛
-> 标准化为 AppMatch
-> predict_match()
-> 生成 PredictionBundle
-> 生成单场推荐
-> 生成二串一推荐
-> 持久化 PredictionSnapshot
-> UI 展示 / 导出报告
```

### 4.2 赛后闭环流程

```text
Titan 抓完场
-> 找到对应快照
-> 生成 SingleSettlement
-> 更新 ELO / 训练样本
-> 自动结算 ParlayTicket
-> 计算 GateMetrics
-> 更新 UI 结算表格
```

## 5. 四个关键缝隙

系统是否稳定，不取决于模块数量，而取决于这四个接口是否干净。

### 5.1 `Data -> Prediction`

输入:

- `AppMatch`

输出:

- `PredictionBundle`

要求:

- 统一对象
- 不允许 UI 直接参与推理

### 5.2 `Prediction -> Recommendation`

输入:

- `PredictionBundle`

输出:

- 单场推荐
- 二串一推荐

要求:

- 推荐只能从标准预测包派生
- 不允许执行层另起一套判断规则

### 5.3 `Recommendation -> Settlement`

输入:

- 推荐快照
- 完场赛果

输出:

- 单场结算
- 二串一结算

要求:

- 每种玩法都有明确命中规则
- 二串一结算必须依赖单场结算，不重复造轮子

### 5.4 `Settlement -> Gate`

输入:

- 单场结算
- 二串一结算

输出:

- `GateMetrics`

要求:

- Gate 只看结果，不看模型内部状态
- 熔断必须可解释

## 8. 当前仍未闭合的边界

这 4 个点是后续开发优先级最高的空隙。

1. 半全场自动结算未闭合
原因: 当前数据源没有半场比分，半全场只能推荐，不能自动判定命中

2. 二串一推荐缺独立管理面板
当前只在详情页附加展示，缺少单独票据视图

3. Gate 仍是统计层，不是调度层
当前能算熔断，但还没有把 `breaker_on` 真正接入“停止出串/停止放行”

4. 二串一候选腿规则仍偏启发式
当前主要按置信度阈值和混合优先做组合，后续应继续加玩法降权和重复暴露控制

## 9. 开发时的硬边界

后续任何改动都尽量不要突破下面三条:

1. 只允许 `predict_match()` 生成标准预测包
2. 只允许结算层生成命中记录
3. 只允许 Gate 基于结算记录做熔断判断

如果这三条被打破，系统会重新回到“逻辑分散、难以闭环”的状态。
